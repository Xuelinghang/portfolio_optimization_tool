During the second sprint, our team made significant progress in implementing key frontend features. We implemented a simple login page with client-side validation using dummy credentials, and a registration page that validates user input while seamlessly integrating with the login feature. We also delivered the portfolio data entry module, which offers manual data entry as well as CSV upload. Our modular JavaScript implementation in the dataEntry.js file handles form submission and CSV parsing successfully, updating the portfolio display dynamically. Uniform styling across all pages—accompanied by a clean header, logo, and responsive layout—ensured that there was a uniform user experience throughout the app.

In the coming week, backend integration is where our focus will shift to. We will be building API endpoints to connect frontend forms with real-time market data sources and provide secure authentication and data storage through the use of a database. We also anticipate enhancing error handling and expanding the utilization of modular code to calculate more financial metrics, such as expected returns and risk measures. Retro of this sprint, we are content with our frontend achievement; however, there were a couple of CSV parsing and form validation issues that required urgent debugging. These life lessons form the foundation of the importance of testing with mock data in the early stages and improved coordination between UI and backend integration to provide us with valuable learnings as we embarked on the next sprint.

After the meeting with the TA, we clarified two things. First, the types of assets that our users can enter depends on the data that our API can fetch, and second, we will merge all asset types into one portfolio and generate one chart instead of having different charts for different types of assets.
