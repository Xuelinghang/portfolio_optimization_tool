During the third sprint, our team shifted its focus from frontend development to backend integration. We successfully built the initial Flask API endpoints that allow the frontend to communicate with the server. These endpoints now handle user registration, login, and portfolio data submission. We also set up a basic SQLite database (users.db) to store user information and asset inputs. In addition, we implemented JWT-based authentication and began integrating placeholder logic to simulate real-time market data fetching.

One of the key technical achievements this week was linking the portfolio entry form—including both manual input and CSV upload—with backend processing. The JavaScript module (dataEntry.js) sends the submitted data via fetch requests to the API, which then validates and stores the assets in the database. We also improved the project’s overall structure by using the Flask application factory pattern, which modularized our routing into separate blueprints (auth, app, and market_data).

While progress was strong, we encountered a few challenges. Some frontend-to-backend communication issues arose, particularly around how asset data was formatted and parsed. We also ran into some CORS-related problems when submitting data to the API, which required Flask configuration updates. Additionally, when testing Sharpe Ratio calculations with mock data, we discovered that unrealistic values could arise, leading us to implement fallback logic during testing.

After meeting with our TA, we clarified that our tool will support multiple asset types—specifically stock, crypto, bond, fund, and ETF. More importantly, we confirmed that assets of different types can be combined into one portfolio and visualized in a single unified chart, allowing users to analyze mixed investment strategies holistically. This feedback helped us better align our design goals with real-world investor needs as we move into the next sprint.
